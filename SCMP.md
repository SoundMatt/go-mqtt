## 1. Purpose and scope

This Software Configuration Management Plan (SCMP) defines how go-mqtt
configuration items are identified, controlled, baselined, and traced. It
applies to all source, tests, requirements, safety artefacts, build scripts, and
the toolchain.

## 2. Configuration items

| Item | Location | Control |
|---|---|---|
| Source and tests | Git repository `SoundMatt/go-mqtt` | Branch + PR review |
| Requirements registry | `.fusa-reqs.json` | Versioned; changes reviewed |
| HARA | `.fusa-hara.json` | Versioned |
| Project config | `.fusa.json`, `.golangci.yml` | Versioned |
| Finding dispositions | `.fusa-dispositions.json` | Reviewer + rationale required |
| Problem reports | `.fusa-problems.json` | Opened/closed via `gofusa pr` |
| Generated evidence | `fmea.*`, `safety-case.*`, `sbom.json`, `provenance.json`, `boundary.mermaid`, `coupling-report.json`, `tara.*` | Regenerated and committed on release |
| Toolchain pins | `go.mod`, CI workflow pins (go-FuSa `v0.30.0`, RELAY `v1.11.0`) | Explicit version pins |

## 3. Configuration identification and baselines

- Every change is a Git commit with a DCO `Signed-off-by` trailer.
- A **baseline** is a `vX.Y.Z` annotated tag applied only to a commit whose CI is
  fully green (see [SVP.md](SVP.md) §6).
- The Software Bill of Materials (`sbom.json`, SPDX) and build provenance
  (`provenance.json`, SLSA) are regenerated for each baseline and committed,
  binding the release to its exact dependency and toolchain set.

## 4. Change control

1. Work proceeds on a `feat/*` or `fix/*` branch off `main`.
2. A pull request targets `main`; CI runs the full gate set.
3. Review is required before squash-merge.
4. `gofusa impact` scopes the requirements/artefacts affected by the change.
5. Safety artefacts are regenerated; traceability MUST remain 100%.

## 5. Baseline integrity

- The release workflow regenerates safety artefacts on each tag and commits them
  with `[skip ci]`, so the committed evidence always reflects the tagged state.
- `gofusa qualify` records an integrity hash of the qualification evidence.
- Provenance and SBOM provide supply-chain integrity (SLSA build track).

## 6. Configuration status accounting

The Software Configuration Index ([generated by `gofusa sci`]) enumerates the
lifecycle data items and their presence. The Git history, tags, and the
per-release evidence commits provide the configuration status record.

## 7. Tool configuration management

The verification toolchain is version-pinned: go-FuSa `v0.30.0` and RELAY
`v1.11.0` in both `.github/workflows/ci.yml` and `.github/workflows/release.yml`,
and the RELAY dependency in `go.mod`. A tool version change is a controlled
change subject to reverification (§4) and re-qualification (`gofusa qualify`).
