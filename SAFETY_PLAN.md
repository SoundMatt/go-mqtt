## 1. Purpose and scope

This Software Safety Plan (SSP) defines the lifecycle, activities, methods, and
responsibilities for the development of go-mqtt
(`github.com/SoundMatt/go-mqtt`) in accordance with:

- ISO 26262:2018 — Road vehicles — Functional Safety (Parts 3, 4, 6, 8)
- IEC 61508:2010 — Functional Safety of E/E/PE Safety-related Systems (Part 3)

go-mqtt is developed as a **Safety Element Out Of Context (SEOOC)** targeting
ASIL-B (ISO 26262) / SIL 2 (IEC 61508). The integrating system is responsible
for system-level HARA, hardware fault model (FMEDA), and allocation.

## 2. Safety requirements

Requirements are annotated in source code with `//fusa:req REQ-XXX-NNN`
comments and managed by [go-FuSa](https://github.com/SoundMatt/go-FuSa).

| ID | Requirement |
|---|---|
| REQ-CLIENT-001 | Client must establish connection to broker within a configurable timeout |
| REQ-CLIENT-002 | Client must handle broker disconnects gracefully and report ErrClosed |
| REQ-CLIENT-003 | Client must be safe for concurrent use from multiple goroutines |
| REQ-PUB-001 | Publish must validate topic is non-empty |
| REQ-PUB-002 | Publish must respect the requested QoS level |
| REQ-SUB-001 | Subscribe must validate topic filter is non-empty |
| REQ-SUB-002 | Subscribe must support MQTT §4.7 wildcard filters ('+' and '#') |
| REQ-SUB-003 | Subscription channel must not block the publishing goroutine |
| REQ-SUB-006 | Subscription channel depth must be configurable |
| REQ-MSG-001 | Message must carry topic and payload |
| REQ-MSG-002 | Message must preserve QoS level |
| REQ-MSG-003 | Message must indicate retained status |
| REQ-MOCK-001 | Mock broker must route messages per MQTT §4.7 |
| REQ-MOCK-002 | Mock broker must be safe for concurrent use |

## 3. Verification

- Unit tests: `go test -race -count=1 ./...`
- Fuzz targets: `FuzzMatchTopic`, `FuzzPublish`
- Integration tests: optional, against Eclipse Mosquitto via Docker
- Static analysis: `golangci-lint`, `go vet`, `gofusa check`
- Coverage target: ≥80% per package
