## 1. Purpose and scope

This Software Verification Plan (SVP) defines the verification activities,
methods, environment, independence, and transition criteria for go-mqtt
(`github.com/SoundMatt/go-mqtt`), developed as a Safety Element Out Of Context
(SEOOC) targeting ASIL-B / SIL 2 / DAL C. It complements the Software Safety
Plan ([SAFETY_PLAN.md](SAFETY_PLAN.md)) and Configuration Management Plan
([SCMP.md](SCMP.md)).

Verification covers the requirement set in `.fusa-reqs.json` (189 atomic
requirements across the functional REQ-* families and the ISO 21434 REQ-SEC
family) and the source under the root module, `mock/`, `v3/`, `v5/`, `broker/`,
and `bridge/*`.

## 2. Verification methods

| Method | Objective | Evidence |
|---|---|---|
| Requirements review | Each requirement is atomic, verifiable, and traced | `gofusa trace` |
| Static analysis | No undefined behaviour, dead code, or coding-standard violations | `gofusa check`, `go vet`, `golangci-lint v2` |
| Unit test | Each requirement is verified by ≥1 test | `gofusa verify`, `go test -race` |
| Structural coverage | Statement coverage measured and reviewed | `gofusa coverage`, `go test -cover` |
| Cybersecurity analysis | No exploitable insecure pattern (gating on ERROR) | `gofusa cyber` |
| Vulnerability scan | No known-vulnerable dependency | `gofusa vuln` |
| Data/control coupling | Inter-module coupling characterised | `gofusa coupling` |
| Fuzzing | Robustness against malformed input | `go test -fuzz` (count-based) |
| Conformance | RELAY spec + behavioural equivalence | `relay conform --strict`, `relay interop --strict` |
| Tool qualification | go-FuSa fit for its verification role | `gofusa qualify` |

## 3. Requirement-to-test traceability

Bidirectional traceability is enforced in CI and MUST be 100%:

- **Forward** — every requirement carries a `//fusa:req` implementation
  annotation and either a `//fusa:test` (functional) or `//fusa:sec-test`
  (security) annotation. `gofusa trace -req-coverage 100` gates the build.
- **Backward** — every annotated source/test location names a requirement that
  exists in `.fusa-reqs.json`.

The current state: 189/189 requirements traced; 180 functional-tested + 9
security-sec-tested = 189 verified; function-annotation density 100%.

## 4. Verification environment

- **Toolchain:** Go 1.25 and 1.26 on ubuntu-latest, macos-latest, windows-latest.
- **Concurrency:** all unit tests run under the Go race detector (`-race`).
- **Integration:** an optional Mosquitto-backed suite (`-tags integration`)
  exercises wire-protocol behaviour against a reference broker; it skips
  gracefully when the broker is unavailable.
- **Determinism:** fuzz runs are count-based (`-fuzztime=Nx`) for reproducibility.

## 5. Independence and review

Verification artefacts are generated by go-FuSa, an independent tool qualified
via `gofusa qualify`. Pull requests are reviewed before merge; CI gates
(test/lint/FuSa lifecycle/conformance) MUST be green before a tag is applied.
Problem reports are tracked in `.fusa-problems.json`.

## 6. Transition criteria

A release tag may be applied only when, on the tagged commit:

1. `go test -race` passes on the full matrix.
2. `gofusa check` reports zero ERROR findings.
3. `gofusa trace -req-coverage 100` passes.
4. `gofusa cyber`, `gofusa vuln`, and `gofusa qualify` pass.
5. `relay conform --strict` and `relay interop --strict` pass.
6. Structural coverage is measured and any uncovered safety-relevant code is
   dispositioned.

## 7. Reverification

Any change to source, requirements, or tools triggers reverification of the
affected items via the same CI gates. `gofusa impact` is used to scope the
affected requirements and artefacts for a given change.
